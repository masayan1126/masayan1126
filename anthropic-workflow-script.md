## フェーズ②：設計（5:30〜7:00）

次は設計です。feature-dev を呼び出します。

（間）

このスキルは内部で複数のサブエージェントが並行稼働します。

（間）

code-explorer が根本原因を特定しました。

（間）

（間）

（間）

## フェーズ④：実装（8:30〜13:00）

実装フェーズです。修正するのは【強調】5つのファイルです。

順番に見ていきます。

（間）

### directoryReader.ts の修正

1つ目、根本修正です。

readDirectory 関数に parentPath というパラメータを追加しました。

（間）

再帰呼び出し時に currentPath を渡すことで、

ファイルのパスが常に【強調】root/dir1/file.md という完全なパスになります。

（間）

### useTabOpen.ts の修正

2つ目、重複チェックのロジックです。

以前は file.path の文字列比較だけでした。

それを——

（間）

isSameEntry() というFile System Access APIのメソッドに置き換えました。

（間）

これはファイルシステムレベルで同一性を判定するので、

シンボリックリンクや別マウントポイントのケースでも正確に動きます。

（間）

ただし、ドラッグ&ドロップ経由のファイルは、

File System Access APIが使えないという制約があります。

（間）

そこでD&Dのフォールバックとして——

【強調】ファイル名・サイズ・最終更新日時の3つを組み合わせた複合キーを使います。

name::size::lastModified という形式で、実質的にユニークな識別子になります。

（間）

エラーハンドリングも丁寧にやっています。

ハンドルが無効化された場合——権限が失効したとか——は debug ログだけ。

予期しないエラーは warn で記録する。

これが後でレビューでも評価されました。

（間）

### TabBar.tsx の修正

3つ目、タブのラベル表示です。

（間）

タブ一覧を走査して、同じ名前が2つ以上あるものだけを duplicateNames に入れます。

useMemo を使っているので、タブが変わった時だけ再計算されます。

（間）

重複してないファイルは今まで通り。

重複してるものだけ【強調】dir1/README.md 形式にする。

シンプルですよね。

（間）

### useTabRestore.ts の修正

4つ目は、少し地味ですが重要な修正です。

（間）

タブを復元する処理に isCancelled というフラグを追加しました。

（間）

コンポーネントがアンマウントされたとき、

非同期で走っていた復元処理が完了してしまうと——

すでに破棄されたコンポーネントに対して状態を更新しようとして、

エラーになることがあります。

（間）

このフラグで【強調】競合状態を事前に防ぎます。

（間）

### useDragDrop.ts の修正

5つ目は、空だった catch ブロックに console.warn を追加して、

エラーが起きたときにログが残るようにしました。

これはレビューで指摘を受ける前に対処した箇所です。

（間）

### ビルドとテスト

5ファイルの修正が終わったら、ビルドとテストを実行します。

npm run build でTypeScriptのコンパイルとViteのビルド——成功。

npm run test で【強調】Vitestの73件のテストが全パス。

（間）

最後にバージョンを 1.2.1 から 1.2.2 に上げて、

zip ファイルを作り直して実装フェーズ完了です。

## フェーズ⑥：レビュー（14:30〜17:00）

次はレビューです。pr-review-toolkit を呼び出すと——

（間）

画面にこんな出力が流れてきます。

「code-reviewer: PR の差分を確認中...」

「type-design-analyzer: 型定義を解析中...」

「silent-failure-hunter: エラーハンドリングをスキャン中...」

（間）

code-reviewer・type-design-analyzer・silent-failure-hunter が【強調】同時に走っています。

人間がレビューするなら順番に見ていくところを、並行でやってくれる。

それぞれ専門の視点で見てくれるんです。

（間）

今回は【強調】3つの問題が見つかりました。

（間）

まず type-design-analyzer から。

（間）

FileInfo 型に dirName フィールドが含まれているが、

これは path から常に導出できる——と指摘されました。

冗長なフィールドを持つと、path と dirName の不整合が発生しうる。

（間）

これ、自分では気づいていなかったんです。

型に dirName を持たせていたんですが、それは path から計算できるので冗長だと。

もし path が変わって dirName を更新し忘れたら不整合が起きる——

というリスクを指摘してくれました。

（間）

2つ目は silent-failure-hunter から。

（間）

useDragDrop.ts の catch ブロックが、すべてのエラーを無音で飲み込んでいる。

getAsFileSystemHandle の失敗と openTab の失敗が区別できない——という指摘でした。

（間）

3つ目は code-reviewer から。

（間）

isSameEntry() の catch ブロックが DOMException を区別していない——という指摘です。

（間）

InvalidStateError や SecurityError のような「想定内のエラー」は debug レベルでいい。

でも、それ以外の予期しないエラーは warn で記録すべきだ——ということです。

（間）

3つの指摘が出てきたところで、自分がタイプしたのは【強調】「1」の1文字だけです。

（間）

「全部直して」という意味で 1 を選ぶ。

（間）

すると——

FileInfo 型と StoredTab 型から dirName フィールドを削除。

getDirName() というユーティリティを fileSystem.ts に追加。

参照しているファイルを全部更新。

isSameEntry() の catch ブロックを DOMException のタイプ別に分岐。

（間）

【強調】一言で全修正が走る。

こういう積み重ねが開発速度の差になってきます。
